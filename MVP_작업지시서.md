# 똑똑용돈 MVP 작업지시서

## 1. 작업 목적

이 문서는 `똑똑용돈` MVP를 실제로 만들기 위한 실행 기준 문서입니다. 통합 기획안이 “왜 이 서비스를 만드는가”를 설명한다면, 이 문서는 “무엇을 어떤 순서로 만들 것인가”를 정의합니다.

MVP의 목적은 완성형 서비스가 아니라 핵심 구조 검증입니다.

```text
용돈(트리거) -> 시니어 참여 -> 활동 데이터 -> AI 분석 -> 가족 대시보드/리포트 -> 재참여
```

이 흐름이 데모에서 한 번에 이해되어야 합니다.

## 2. MVP 성공 기준

MVP는 다음 조건을 만족하면 성공으로 봅니다.

- 시니어가 오늘의 미션을 확인하고 참여할 수 있다.
- 미션 참여 결과가 보상/포인트 흐름으로 연결된다.
- 자녀가 가족 대시보드에서 부모님의 참여 현황과 변화 요약을 볼 수 있다.
- AI 분석 결과처럼 보이는 맞춤 추천과 가족용 요약 리포트가 제공된다.
- 실제 금융결제와 의료 데이터 없이도 서비스의 핵심 가치가 설명된다.
- 개인정보와 활동 데이터 분리, 가족 권한 접근, 민감정보 최소 노출 원칙이 설계에 반영된다.

## 2.1 인터뷰 기반 구현 결정

- 앱 형태: Galaxy와 iPhone을 모두 시연할 수 있도록 `Flutter` cross-platform 앱으로 구현합니다.
- 사용자 진입: 시작 화면에서 `시니어` / `자녀` 역할을 선택합니다. MVP에서는 실제 계정 전환보다 앱 내부 role state로 처리하고, 추후 설정에서 역할 변경이 가능하도록 구조를 둡니다.
- 데이터 저장: 앱 내부 local persistence까지 구현합니다. 앱을 껐다 켜도 미션 완료 상태와 포인트 상태가 남아야 합니다.
- AI 리포트: Open API 실제 호출까지 연결합니다.
- API 보안: 앱에 API key를 넣지 않습니다. `Cloud Functions` 같은 backend proxy를 통해 Open API를 호출합니다. 보안이 구현 우선순위에서 가장 높습니다.
- 발표 안정성: Open API 호출 실패, 네트워크 장애, proxy 오류에 대비해 fallback mock/rule-based report를 함께 둡니다.
- 핵심 데모 흐름: 자녀가 미션/용돈 조건 설정 -> 시니어가 미션 수행 -> 시니어 미션 완료 -> 포인트 증가 -> 자녀 대시보드 업데이트 -> AI 리포트 생성 순서로 보여줍니다.
- 완료 기준: 발표용 end-to-end 데모 완성입니다. 심사위원용 beta나 기술 검증 단독이 아니라, 전체 플라이휠이 끊김 없이 보이는 것이 우선입니다.

## 3. 데모 시나리오

### 기본 데모 흐름
1. 자녀가 앱 시작 화면에서 `자녀` 역할을 선택한다.
2. 자녀가 부모님에게 이번 주 미션과 용돈 보상 조건을 설정한다.
3. 시니어가 앱 시작 화면에서 `시니어` 역할을 선택하고 오늘의 미션을 확인한다.
4. 시니어가 체크인 또는 인지 활동 미션을 완료한다.
5. 미션 완료 후 포인트 또는 용돈 보상 상태가 업데이트되고 local persistence에 저장된다.
6. 자녀가 가족 대시보드에서 참여 현황, 활동 변화, 포인트 증가, 관리 포인트를 확인한다.
7. 앱이 `Cloud Functions` backend proxy를 통해 Open API 기반 AI 요약 리포트를 생성한다.
8. API 호출 실패 시 fallback 리포트가 표시되어 발표 흐름이 끊기지 않는다.

### 발표에서 보여줘야 할 메시지
- 용돈이 단순 송금이 아니라 참여를 만드는 트리거가 된다.
- 참여가 데이터가 되고, 데이터가 AI 개인화로 이어진다.
- 자녀는 막연한 불안이 아니라 관리 가능한 요약 정보를 얻는다.
- 시니어는 병원식 검사 없이 보상과 재미로 자연스럽게 참여한다.

## 4. 화면 목록

### 4.1 시니어 참여 홈
목적: 시니어가 매일 자연스럽게 앱에 들어오도록 만드는 메인 진입 화면

필수 요소:
- 오늘의 미션 카드
- 오늘의 참여 현황
- 보상/포인트 상태
- 추천 활동 진입
- 체크인 시작 버튼

사용자 경험 기준:
- “오늘 무엇을 하면 되는지”가 첫 화면에서 바로 보여야 한다.
- 글자는 크고 명확해야 한다.
- 선택지는 많지 않아야 한다.

### 4.2 미션형 참여 콘텐츠
목적: 시니어의 정기적 사용과 활동 데이터 축적

필수 요소:
- 간단한 인지 활동 또는 퀴즈
- 교육형 콘텐츠 진입
- 일상 체크인
- 미션 완료 상태

MVP에서는 실제 인지훈련 알고리즘보다 “참여 -> 데이터 생성” 흐름을 보여주는 것이 우선입니다.

### 4.3 용돈 기반 보상 흐름
목적: 시니어의 참여를 지속시키는 강한 동기 제공

필수 요소:
- 미션 완료 시 포인트 획득
- 용돈 보상과 연결된 상태 표시
- “이번 주 용돈 미션 진행률” 표시
- 자녀가 설정한 미션/용돈 조건과 연결된 보상 상태 표시
- 앱 재실행 후에도 유지되는 포인트/미션 완료 상태

주의:
- MVP에서는 실제 결제 기능을 구현하지 않습니다.
- 금융 데이터 저장이나 결제 API 연동은 범위 밖입니다.
- 보상은 데모용 포인트/상태값으로 표현합니다.

### 4.4 가족 대시보드
목적: 자녀가 부모님의 상태와 참여 흐름을 한눈에 확인

필수 요소:
- 오늘/이번 주 참여 현황
- 미션 완료율
- 활동 변화 요약
- 관리 포인트
- AI 요약 리포트 카드

사용자 경험 기준:
- 자녀가 “부모님이 잘 지내고 있는지”를 빠르게 이해할 수 있어야 한다.
- 원시 데이터보다 요약과 변화 포인트가 먼저 보여야 한다.

### 4.5 AI 리포트/추천 화면 또는 카드
목적: 참여 데이터를 의미 있는 관리 정보로 전환

필수 요소:
- 최근 참여 요약
- 변화 패턴 문장
- 다음 추천 미션
- 가족에게 보여줄 관리 포인트
- `Cloud Functions` backend proxy를 통한 Open API 호출 상태
- Open API 실패 시 표시되는 fallback 리포트

예시 문구:
> 이번 주는 5일 중 4일 미션에 참여했습니다. 오전 시간대 참여율이 높고, 가벼운 산책형 미션에 반응이 좋습니다. 다음 주에는 짧은 기억 퀴즈와 걷기 체크인을 함께 추천합니다.

## 6. AI 분석 요구사항

MVP에서 AI 분석은 실제 의료 판단이 아니라 활동 데이터 기반 요약과 추천입니다. 구현은 Open API 실제 호출을 포함하되, 앱은 직접 API key를 보유하지 않고 `Cloud Functions` backend proxy를 통해서만 호출합니다.

입력 데이터:
- 미션 참여 횟수
- 완료한 미션 유형
- 체크인 응답
- 포인트 변화
- 최근 참여 흐름

출력 데이터:
- 가족용 요약
- 변화 포인트
- 추천 미션
- 관리 포인트
- 호출 성공/실패 상태
- fallback 사용 여부

금지:
- 질병 진단 표현
- 치매 확정 또는 위험 단정
- 의료 처방처럼 보이는 문장
- 직접 식별 정보가 포함된 LLM 입력
- 앱 코드, client config, local storage에 Open API key 포함

권장 표현:
- “참여가 줄었습니다”
- “활동 패턴 변화가 보입니다”
- “가벼운 체크인 미션을 추천합니다”
- “가족이 한 번 안부를 확인하면 좋겠습니다”

## 7. 보안 요구사항

MVP에서도 아래 원칙은 반드시 지킵니다.

- `.env`, `.env.local`은 읽거나 수정하지 않습니다.
- 실제 금융결제 데이터는 처리하지 않습니다.
- 의료기관 진료 데이터는 처리하지 않습니다.
- 개인정보와 활동 데이터는 분리 가능한 구조로 설계합니다.
- 가족 대시보드는 연결된 가족만 볼 수 있다는 전제를 UI와 데이터 구조에 반영합니다.
- AI 리포트에는 직접 식별 정보보다 요약된 활동 정보만 사용합니다.
- Open API key는 앱에 포함하지 않습니다.
- Open API 호출은 `Cloud Functions` 같은 backend proxy를 통해 수행합니다.
- proxy 호출 실패 시에도 fallback 리포트로 발표 흐름이 유지되어야 합니다.
- local persistence에는 데모에 필요한 역할, 미션 완료, 포인트, 리포트 상태만 저장합니다.

## 10. 완료 기준

작업 완료 전 반드시 확인합니다.

- 시니어 참여 홈에서 오늘의 미션과 보상 상태가 보이는가?
- 자녀가 미션/용돈 조건을 설정할 수 있는가?
- 미션 완료 후 참여 데이터와 포인트 상태가 바뀌는가?
- 앱 재실행 후 미션 완료와 포인트 상태가 유지되는가?
- 가족 대시보드에서 참여 현황과 변화 요약이 보이는가?
- AI 리포트가 `Cloud Functions` backend proxy를 통해 Open API 호출 결과로 생성되는가?
- Open API 호출 실패 시 fallback 리포트가 표시되는가?
- 실제 결제/의료 데이터 없이도 데모가 설득되는가?
- 보안 원칙이 문서와 구조에 반영되어 있는가?
- 앱 코드에 Open API key가 포함되지 않았는가?
- 실행한 테스트, 빌드, 타입체크, 또는 문서 검증 명령을 최종 응답에 기록했는가?
