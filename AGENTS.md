# AGENTS.md

## 프로젝트 개요

- `똑똑용돈` 프로젝트의 MVP 제작 공간이다.
- 핵심 아이디어는 부모님께 드리는 용돈을 시니어의 미션 참여, 활동 데이터, AI 개인화 케어, 가족 대시보드로 연결하는 것이다.

```text
용돈(트리거) -> 시니어 참여 -> 활동 데이터 -> AI 분석 -> 개인화 -> 재참여
```

- 현재 저장소는 구현 코드 저장소가 아니라 문서 중심의 기획/작업지시 저장소다.

## 현재 파일 구조

```text

├── AGENTS.md                          # 에이전트 작업 규칙과 프로젝트 지식
├── HANDOFF.md                         # 다음 에이전트용 인수인계 문서
├── MVP_작업지시서.md                  # MVP 구현 계획의 기준 문서
├── 코드게이트_똑똑용돈 통합 기획안.md  # 통합 기획/제출 기준 문서
```

## 먼저 읽을 문서

| 목적 | 읽을 문서 | 메모 |
| --- | --- | --- |
| 현재 상태 파악 | `HANDOFF.md` | 이 파일 다음으로 읽는다. |
| 기획안/제출 문장 수정 | `코드게이트_똑똑용돈 통합 기획안.md` | 현재 기준 기획안이다. |
| MVP 구현 계획 수립 | `MVP_작업지시서.md` | 화면, 데이터, AI, 보안, 검증 기준이 들어 있다. |
| 이전 표현 비교 | `코드게이트_똑똑용돈 기획안.md`, `코드게이트_똑똑용돈 기획안 상세.md` | 참고용이다. 요청 없이 삭제/이동하지 않는다. |

## 작업 방식

- 이 폴더에서는 `superpowers` 플러그인 중심으로 작업한다.
- LazyCodex/OmO 계열의 스킬, 하네스, 작업 구조는 이 프로젝트의 기본 작업 방식으로 사용하지 않는다.
- 작업 전 한 줄로 가정을 밝히고 바로 진행한다.
- 해석이 여러 개면 가장 가능성 높은 방향을 선택하고, 선택한 해석을 말한다.
- 질문은 결과물이 달라지고 잘못 선택했을 때 되돌리기 어려운 경우에만 한다.
- 사용자는 승인 루프보다 진행을 선호한다. 작은 가정은 말하고 진행한다.
- 변경은 요청에 필요한 파일에만 한다.
- 관련 없는 문서, 초안, 히스토리는 건드리지 않는다.


## 코드/구현 작업 원칙

- 최소한의 코드로 요청을 해결한다.
- 요청받지 않은 기능, 추상화, 설정 가능성은 추가하지 않는다.
- 기존 구조와 스타일을 우선 따른다.
- 새 SQL 쿼리나 DB 함수를 만들기 전에는 기존 디렉토리 구조와 중복 로직을 확인한다.
- 쿼리나 DB 함수를 추가/수정한 경우 Query Duplication Prevention Hook 관련 검증에 위배되지 않는지 확인한다.
- 리뷰 자동화 스크립트나 Hook 관련 코드는 지정된 디렉토리 하위에서만 작업한다.

## 수정 금지

- `.env`, `.env.local`은 절대 읽거나 수정하거나 출력하거나 요약하지 않는다.
- 기존 데이터베이스 마이그레이션 내역은 명시 요청 없이 수정하거나 삭제하지 않는다.
- 원본 기획안 파일은 사용자가 요청하기 전까지 삭제하거나 이동하지 않는다.

## 보안 및 데이터 경계

- MVP에서는 실제 금융결제 데이터를 처리하지 않는다.
- MVP에서는 의료기관 진료 데이터나 과도한 의료 데이터를 처리하지 않는다.
- AI 리포트는 질병 진단, 확정적 위험 판단, 처방처럼 보이는 표현을 피한다.
- AI 리포트는 활동 요약, 변화 포인트, 관리 제안 중심으로 작성한다.
- 가족 대시보드는 연결된 가족만 볼 수 있다는 전제를 UI와 데이터 구조에 반영한다.
- 개인정보와 활동 데이터는 분리 가능한 구조로 설계한다.

## MVP 핵심 범위

- 시니어 참여 홈
- 교육형·미션형 콘텐츠
- 실제 결제가 아닌 데모용 포인트/보상 흐름
- 가족 대시보드
- AI 추천 및 요약 리포트
- 계정 연결, 접근 권한, 개인정보/활동 데이터 분리 원칙

## 검증 규칙

- 완료했다고 말하기 전에 지정된 로컬 검증을 실행한다.
- 문서만 수정한 경우 파일 존재와 UTF-8 읽기를 확인한다.
- 코드 변경 시 가장 작은 관련 검증부터 실행한다.
- 테스트 구성이 없으면 빌드, 타입체크, 린트, 또는 파일 수준 sanity check를 실행한다.
- UI 변경 시 실제 앱 화면이나 라우트를 확인한 뒤 보고한다.
- 최종 응답에는 변경 파일과 실행한 검증 명령/결과를 반드시 적는다.
